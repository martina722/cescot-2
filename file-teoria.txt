Interazione con DBMS
Quando si realizza un'applicazione (ogni tipo) è necessario un sistema in grado di memorizzare dati che vengono gestiti dall'applicazione (anagrafiche, documenti, ...), questo tipo di tecnica consente agli utenti di inserire, modificare e visualizzare i dati necessari. I  DBMS consentono di gestire grandi quantità di dati e dati strutturati (ma anche destrutturati), fornendo strumenti per leggere e scrivere i dati (CRUD). Per realizzare una pagina dinamica è necessario interfacciare il linguaggio utilizzato per generare la pagina (PHP, C#, ...) con il DBMS, questo richiede un dialogo tra server (server diversi: server web e DBMS). Il dialogo fra server web e DBMS avviene tramite API. il server web invia richieste al server DBMS tramite le API che questo deve mettere a disposizione. PHP offre diverse tecniche di interfacciamento con i DBMS, le principali sono:
- PDO, è generico e consente di interagire con diversi tipi di DBMS, è più complesso da utilizzare ma versatile
- MySQLi, consente di interagire solamente con MySQL (o altri DBMS che implementino le stesse API)
PDO è disponibile solamente come classe, pertanto richiede un approccio basato su programmazione ad oggetti, mentre MySQLi è disponibile sia come classe che come funzione, pertanto può essere utilizzato sia con programmazione ad oggetti che tramite funzioni.
MySQLi
Per utilizzare una tabella di database all'interno di una pagina web è necessario:
- connettersi al database (come farei con PHPMyAdmin, che si collega al server internamente)
- eseguire le query necessarie
- costruire il risultato da mostrare nella pagina in base al risultato (una variabile strutturata, doppio array) fornito dalla query
Per connettersi al database è necessario utilizzare MySQLi, passando le informazioni (argomenti) necessari alla connessione (indirizzo del server, nome del database, username, password). MySQLi restituisce una variabile che contiene la connessione al database, grazie a questa variabile è possibile eseguire query.
La funzione mysqli_connect si collega al database e restituisce una variabile che contiene la connessione. La funzione mysqli_query esegue una query e restituisce una variabile che contiene il risultato della query. La funzione fetch consente di ciclare all'interno della tabella contenuta nel risultato della query, restituisce un array che contiene il valore di ogni campo della tabella.
Rischi dell'interazione con database
Quando si lavora con i database, è importante  tenere in considerazione i rischi legati all'interazione fra i linguaggi utilizzati sul server web e il DBMS. Il rischio più comune che si corre è quello della SQL Injection, una tecnica basata su SQL che può consentire ad utenti non autorizzati di eseguire query a loro piacimento sul database. Gli utenti possono interagire con il database tramite l'applicazione messa a loro disposizione, compilando moduli o modificando i parametri presenti sull'URL delle pagine. Se non si controlla che le variabili che vengono utilizzate per costruire le query siano "sicure", si rischia che gli utenti possano inserire intere query all'interno di un parametro dell'URL o di un campo di un modulo e che la nostra procedura le esegua.


JOIN
Le JOIN consentono di ricostruire le relazioni fra tabelle, tipiche dei database relazionali. Le tabelle hanno relazioni strutturali, queste relazioni sono evidenziate nel diagramma E/R (entities relationship) associato al databaase.
Le JOIN derivano direttamente dalle relazioni presenti nel diagramma E/R e sono costituite dalle corrispondenze fra i campi che mettono in relazione le tabelle. Le join non dipendono dalla query, ma solamente dal diagramma E/R, es: join tra clienti e prenotazioni.
clienti INNER JOIN prenotazioni ON clienti.id_cliente = prenotazioni.cliente
Le join possono essere di 4 tipi:
- INNER JOIN, restituisce solamente i record che hanno una corrispondenza fra le due tabelle
- LEFT OUTER JOIN, restituisce tutti i record della tabella di sinistra e i soli record della tabella di destra che hanno una corrispondenza nella tabella di sinistra
- RIGHT OUTER JOIN, restituisce tutti i record della tabella di destra e i soli record della tabella di sinistra che hanno una corrispondenza nella tabella di destra
- FULL OUTER JOIN (non supportata da MySQL), restituisce tutti i record di entrambe le tabelle
Nelle OUTER JOIN, gli eventuali record che non hanno corrispondenza vengono riempiti con valori NULL
clienti LEFT OUTER JOIN prenotazioni ON clienti.id_cliente = prenotazioni.cliente
Es. restituire nome della città, nome e cognome del cliente, data di arrivo e data di partenza per tutte le prenotazioni che hanno importo superiore a 250.
SELECT citt.citta,clienti.nome,clienti.cognome,prenotazioni.arrivo,prenotazioni.partenza
FROM citta
INNER JOIN clienti ON citta.id_citta = clienti.citta
INNER JOIN prenotazioni ON clienti.id_cliente = prenotazioni.cliente
WHERE prenotazioni.importo > 250

Lavorare come libero professionista
Le aziende di sviluppo tendono a limitare il numero di dipendenti e collaboratori (lavorano solo quando c'è necessità, sono free lance) per limitare i costi. Nel settore dello sviluppo software è diffusa la prassi, da parte delle software house, di affidare parte del lavoro a professionisti esterni, questo è dovuto a:
- mancanza di competenze in azienda (condizione tipica in questo settore, non è un limite)
- fluttuazioni del lavoro
Il libero professionista per sua natura non ha vincoli ne di tempo ne di orari ne di esclusiva (un titolare non può dire ad es. ad un libero professionista che per mesi devi lavorare solo da me, è illegittimo, perchè è una specie di assunzione senza essere assunti), ma non ha garanzie occupazionali (se l'azienda dice che per 3 mesi lavorerai da lei e il progetto è saltato, in caso si chiuda il contratto c'è una penale poichè hai rinunciato ad altre occupazioni per lavorare in quell'azienda). In molti casi, soprattutto tra i profili medio-alti, si firmano accordi NDA (Non-Disclosure Agreement), patti di non concorrenza che hanno lo scopo di vincolare il lavoro futuro del libero professionista, evitando concorrenza con l'azienda che gli ha fornito il lavoro

Tipologie societarie
Il libero professionista deve configurarsi come azienda, questo implica il possesso di una PI, la quale può essere personale o societaria, le partite iva personali hanno semplificazioni burocratiche e bassi costi di gestione, mentre quelle societarie hanno burocrazie più complesse e costi fissi. Le partite iva personali sono:
- forfetaria, ha limite di fatturato, tassazione agevolata, non può scaricare (detrarre) spese, ne personali ne professionali
- ordinaria, non ha limite di fatturato, può scaricare (detrarre) spese professionali

Compensi 
I liberi professionisti devono valutare i prezzi che applicano per tenere conto del loro rischio d'impresa:
- malattia
- mancanza di commesse
- ferie non pagate
Per compensare il rischio d'impresa, è necessario avere costi più alti rispetto a lavoratori dipendenti che abbiano lo stesso profilo, in base a quest'ultimo i compensi partono da piò o meno il doppio di quello che è il costo di un dipendente. Mediamente, i prezzi per i free lance partono da €40 + iva all'ora.

Autopromozione
Per ottenere clenti è necessario farsi conoscere dalle aziende, questo richiede l'utilizzo di tecniche di branding e di promozione personale. Le tecniche più comuni sono:
- contatto diretto con le aziende, ottenuto tramite lavori precedenti, eventi, rete di contatti
- digital marketing, utilizzando siti web proprietari, social network.

Aggregazione
Il linguaggio SQL prevede l'utilizzo di funzioni di aggregazione, le principali sono:
- COUNT(*), conta i record
- SUM(<campo>), calcola il totale dei valori presenti nel campo, funziona solamente su campi numerici
- AVG(<campo>), calcola la media dei valori presenti nel campo, funziona solamente su campi numerici
- MIN(<campo>), restituisce il minimo dei valori presenti nel campo, funziona su qualsiasi tipo di dato
- MAX(<campo>), restituisce il massimo dei valori presenti nel campo, funziona su qualsiasi tipo di dato
SELECT DISTINCT camera
FROM prenotazioni
ORDER BY camera ASC
Le funzioni di aggregazione riducono le righe di una tabella risultante dalla query applicando funzioni matematiche ai dati.
SELECT SUM(importo)
FROM prenotazioni
Questa query restituisce sempre un record, indipendentemente dal numero di recordo presenti in tabella, SUM calcola la somma dei valori presenti nel campo, restituisce un solo record.

GROUP BY
La clausola GROUP BY raggruppa i dati della tabella in base ai campi indicati, per ogni gruppo calcola le funzioni di aggregazione. Le funzioni di aggregazione non possono essere utilizzate all'interno di una SELECT assieme a campi, a meno che tutti i campi presenti nella SELECT siano presenti anche nella GROUP BY.
SELECT tipo_struttura, SUM(importo) AS totale
FROM prenotazioni
GROUP BY tipo_stuttura
All'interno della SELECT di una query è possibile utilizzare più funzioni di aggregazione contemporaneamente, come se fossero normali campi.
SELECT tipo_struttura, --> indica il criterio di raggruppamento (es. hotel, b&b, appartamento)
SUM(importo) AS totale_fattura, --> somma tutti gli importi per ogni tipo di struttura
SUM(caparra) AS totale_caparra, --> somma le caparre per ogni tipo di struttura
AVG(importo) AS importo_medio --> calcola la media degli importi per ogni tipo di struttura
FROM prenotazioni --> indica la tabella dove prendere i dati, significa che tutte le colonne usate nel SELECT (tipo_struttura, importo, caparra) devono esistere nella tabella prenotazioni
GROUP BY tipo_struttura --> raggruppa tutte le righe che hanno lo stesso tipo di struttura; ogni funzione aggregata (SUM, AVG) viene calcolata per gruppo
L’ordine logico di esecuzione di una query SQL è questo:
1- FROM
2- HERE
3- GROUP BY
4- HAVING
5- SELECT
6- ORDER BY
Quando si scrivono query all'interno del codice di applicazioni, è necessario che tutti i campi restituiti abbiano un nome univoco. Per sviluppare le query si possono utilizzare ambienti di lavoro appositi (phpmyadmin, heidiSQL, ...), in modo da semplificare la scrittura delle query, questi ambienti non hanno gli stessi vincoli dei linguaggi di programmazione. Quando si scrive una query con un ambiente apposito, e la si importa all'interno del linguaggio di programmazione, possono nascere errori dovuti a nomi campo assenti (funzioni che generano campi) o duplicati, in questi casi è sufficiente utilizzare alias (AS) per verificare i campi senza nome.

Interazione GUI (interfacce grafiche)/database
Nelle applicazioni, gli utenti lavorano con le interfacce grafiche (GUI), queste però mostrano dati memorizzati e mandati dal database. Le query che vengono eseguite dal software possono contenere delle parti variabili, queste parti vengono compilate in base agli input forniti dagli utenti.

$query = 'SELECT clienti.nome, clienti.cognome, regioni.regione, regioni.area_geografica, citta.citta
    FROM clienti
    INNER JOIN citta ON clienti.citta = citta.id_citta
    INNER JOIN regioni ON citta.regione = regioni.regione';
    WHERE regioni.regione = "' . $regione_da_cercare . '"';

Quando si utilizzano query dinamiche, è sufficiente assegnare i valori corretti alle variabili che rendono dinamiche le query per caricare i dati che interessano all'utente. Le variabili devono essere valorizzate all'interno del codice PHP, mentre l'utente utilizza le pagine HTML; per consentire alle pagine HTML di passare dati alle procedure PHP si possono utilizzare i metodi GET e POST. Tramite form o link è possibile passare parametri ad una pagina.
